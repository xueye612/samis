# 后端现状与护理系统合并开发说明

生成时间：2026-06-02
依据：
- 现状核对：huliTp6System-master（25 个 DDD 业务域 + 49 张表 + samis 应用骨架）、shuhu-surgianes/backend-tp6（5 个 phase migration）
- 主源文档：docs/03_dictionary/database-tables.md、docs/04_delivery/current-implemented-apis.md、docs/05_adr/0001-nursing-sot-integration.md
- 决策：mazui/项目说明.txt（明确麻醉模块挂在 huliTp6System-master/samis/，API 前缀 /api/samis/pc/v1/）

---

## 1. 当前后端基础是什么

存在三套后端代码：

| 仓库 | 角色 | 现状 |
|---|---|---|
| huliTp6System-master | 手术护理系统后端（ThinkPHP 6 + DDD） | 主线生产代码：25 个 domain、49 张业务表、内部含 samis/ 麻醉应用骨架（仅记录单 6 个表 + 4 个集成网关） |
| shuhu-surgianes/backend-tp6 | 麻醉独立验证后端（ThinkPHP 6） | 已落地 sys_ + surg_ + anes_ 全套 migration（5 个 phase），含 PACU / 抢救 / 访视 / 字段丰富 |
| samis-web | 麻醉前端（Vue3） | 调用护理登录 + `/api/samis/pc/v1/*` 麻醉接口 |

> 结论：当前**事实上存在两份麻醉后端**。shuhu-surgianes/backend-tp6 是“能力最完整的麻醉后端实现”，huliTp6System-master/samis 是“与护理系统并存的最终落地形态”。

---

## 2. 是否应继续沿用现有护理系统后端

**应当沿用，不另起一套独立后端。**

理由：
1. 护理后端已有 25 个 DDD 业务域和 49 张表，覆盖患者/手术/排班/人员/字典/权限/日志/输血/交接/核查/器械/评估等基础与共用业务。
2. 麻醉系统的患者、手术、人员、手术间、字典、权限本就和护理共用，重写一遍是无意义的复制。
3. 护理后端已建立 samis 应用骨架与 4 个只读集成网关，是最低成本的接入方式。
4. 麻醉端独立后端的 sys_ / surg_ 表已经在 shuhu-surgianes/backend-tp6 中验证，可作为“向护理合并时的结构参考”。

最终形态：
- 共享底座：护理后端 huliTp6System-master 提供基础能力。
- 麻醉应用：huliTp6System-master/samis/ 承载麻醉业务（Controller / Service / Repository / Model）。
- 麻醉接口前缀：/api/samis/pc/v1/。
- 数据库：与护理同服务器，麻醉独立 schema（samis 库），通过引用键关联护理库。

## 3. 哪些基础表可以复用（来自护理）

复用规则：**只读引用，不复刻**。

| 业务域 | 护理主表（SoT） | 麻醉端使用方式 |
|---|---|---|
| 用户 / 人员 | user_information、admin_user、admin_user_group、admin_category | 通过 sys_user 适配层 或 集成网关读取 |
| 角色权限 / 菜单 | powerlist、admin_category | 沿用护理权限模型，麻醉新增 `anes:*` 权限点 |
| 字典 / 选项 | option_cat、form_content_dictionary、form_content_dictionary_options | 仅读 + 新增麻醉专用字典分类 |
| 患者 | operatenotice（含患者字段） | 仅读 + 必要时建快照（不另起患者主表） |
| 手术通知 / 主数据 | operatenotice、operatenotice_ext | 仅读，通过 OPERATIONID 关联 |
| 手术间 | operation_room、operation_room_type | 仅读 |
| 排班 | schedule | 仅读 + 麻醉端可在 anes_record / surg_operation_staff 中补充人员 |
| 患者交接 | handover_send、handover_receive_restore | 仅读 + 角色补签（HuliHandoverGateway） |
| 安全核查 | security、security_vice | 仅读 + 麻醉端可补签字段（HuliSecurityGateway） |
| 输血 | xue、blood_confirm、v_sm_pda_blood | 仅读（HuliBloodGateway），麻醉端在 anes_record 中保存自身视角的输血明细 |
| 用药执行 | medicant_enforcement、medicant、narcotic_enforcement | 仅读 / 联动；麻醉用药主表独立 |
| 评估 | evaluation_hypothermia、evaluation_pressure_damage 等 | 仅读，结论在 anes_record 中引用 |
| 器械清点 | apparatus、apparatus_check_package(_detail) 等 | 仅读 |
| 访视 | before_operation_visit、after_operation_visit | 仅读；麻醉独立维护 anes_pre_visit / anes_post_visit |
| 签名 | signature | 复用，麻醉签名记录到自有签名表 |
| 日志 / 审计 | 中间件 + legacy | 复用同一审计体系 |

## 5. API 应该如何按模块拆分

固定前缀：`/api/samis/pc/v1/`。

控制器划分（与 docs/02_specs/backend-apis.md 一致）：

| 控制器 | 说明 |
|---|---|
| Auth | 沿用护理登录或对接 |
| System | 麻醉端字典、模板、设备、接口配置（涉及修改类） |
| Surgery | 共享手术主数据的麻醉视角（如今日、详情、关键时间） |
| Anesthesia | 访视、同意书、核查（仅签名）、记录、小结、术后访视 |
| Pacu | PACU 主记录、体征、评估、用药、入出量、计费 |
| Quality | 工作量、运营指标、26 项质控（二期） |
| MedicalRecord | 文书 PDF / 签名 / 归档 / 提交日志（与 mr_* 表绑定） |
| Billing | 麻醉与 PACU 计费 |
| Interfaces | 外部 HIS / EMR / LIS / PACS / 病理 / 设备 |
| Mobile | 移动端聚合接口 |
| Nursing | **只读聚合**：/api/nursing/<domain>/list|detail（domain：user/dict/patient/operation/schedule/evaluation/medicant/handover/security） |

> 已实现的麻醉接口（节选）：
> - /api/anes/record/list|detail|init|save|sign|unsign|lock|archive
> - /api/anes/visit/list|detail|save|delete|sign|lock|archive（kind=pre/post）
> - /api/anes/vital/batch-save
> - /api/pacu/*（含 Aldrete / 入出量 / 用药）

## 6. 权限、日志、字典、人员、菜单如何复用

| 能力 | 复用方式 |
|---|---|
| 权限 | 沿用护理 powerlist 模型，统一通过 sys_role + sys_permission 表达，麻醉端只新增 `anes:*` 权限点 |
| 菜单 | 麻醉菜单挂在统一菜单树（sys_menu），由 `/api/auth/menu` 下发；菜单项绑定权限点 |
| 字典 | sys_dict_type / sys_dict_item 为主入口；麻醉特有字典作为新分类 |
| 人员 | sys_user 与护理 user_information 单点对齐（同源同步或共表）；麻醉端通过 sys_user.id 关联 |
| 操作日志 | 沿用同一审计中间件（AuditLog），写入 sys_audit_log |
| 登录日志 | 护理已有，麻醉端不另起 |
| 接口日志 | sys_interface_log；外部接口回调统一记录 |

## 7. 麻醉业务数据如何关联手术通知、患者、手术间、医护人员

所有 `anes_*` 业务表必须保留以下关联键：

- `operation_id`：关联 OPERATIONID（护理 operatenotice 或共用 surg_operation）
- `patient_id`：关联 surg_patient（与护理患者字段映射）
- `operation_room_id`：关联 surg_room（与护理 operation_room 映射）
- `staff_id` 系列字段：关联 sys_user（与护理 user_information 映射）

新增 `anes_nursing_shared_ref` 用于登记跨库引用（reference / snapshot / sign-on / extract 四种关系），避免散落字段引发的歧义。

## 8. 后续是否应该独立数据库，还是在现有护理数据库中扩展表

**采用“同服务器双库”方案：**

- 护理库：保留护理原生表与共用主表
- 麻醉库（samis schema）：放置所有 `anes_*` / `qc_*` / `bill_*` / `ai_*` 业务表
- 共用表（`sys_*` / `surg_*` / `mr_*`）：作为基础平台层，归属护理库或独立 platform 库，麻醉库通过权威接口或视图访问

理由：
- 麻醉业务表结构与护理风格差异较大（命名前缀、状态机、签名模式），混入护理库会污染既有字典
- 双库便于灰度、备份与独立升级
- 已在 mazui/项目说明.txt 中明确：“两个数据库会在一个服务器上，患者信息公用”
- samis/sql/00_init_samis.sql 已经按 samis 库初始化

## 9. 与现有手术护理系统合并开发时应避免哪些重复建设

> 这部分作为合并开发时的红线清单：

1. 不要在麻醉端重新做登录 / 权限 / 菜单 / 字典 / 人员 / 科室。
2. 不要在麻醉端重建患者主表、手术主表、手术间、排班主表。
3. 不要在麻醉端重建患者交接、安全核查、器械清点的主表。
4. 不要在麻醉端重建输血主表（仅可在 anes_record 中保留输血明细字段）。
5. 不要在麻醉端复刻护理低体温 / 压力损伤 / 风险评估表。
6. 不要在麻醉前端直接对接护理业务表，统一走 `/api/nursing/<domain>/*` 与集成网关。
7. 不要在麻醉端做独立的护理 SoT 字段同名表，统一通过 `anes_nursing_shared_ref` 引用。
8. 不要让 `app\*`（护理）反向依赖 `samis\*`（麻醉），保持单向依赖。

---

## 10. 结论速览

- 不建议另起一套完全独立后端。
- 建议在现有手术护理系统后端基础上扩展 anesthesia 业务域，落地在 huliTp6System-master/samis/，API 前缀 /api/samis/pc/v1/。
- 共用 system / surgery / patient / staff / dictionary / log 等基础能力。
- 新增 anes_record / anes_pre_visit / anes_post_visit / anes_event / anes_medication / anes_vital_sign / anes_pacu_* / anes_rescue_* / qc_* / bill_* 等麻醉业务表。
- 所有新增表必须保留 operation_id、patient_id、operation_room_id、staff_id 关联。
- shuhu-surgianes/backend-tp6 的 migration 与 service 层是现成的结构参考，向 huliTp6System-master/samis/ 合并时直接采纳即可。
