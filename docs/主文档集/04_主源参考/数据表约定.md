## 表前缀

| 前缀 | 说明 | 是否预留与护理系统共用 |
|---|---|---|
| `sys_*` | 平台基础（用户/角色/权限/字典/日志/接口配置）| ✓ 共用 |
| `surg_*` | 手术主数据（申请/排台/手术/通知/取消）| ✓ 共用 |
| `anes_*` | 麻醉业务（访视/核查/记录/小结/PACU）| 麻醉独有 |
| `bill_*` | 计费 | 麻醉独有 |
| `mr_*` | 电子病历 / 文书中心 | 共用 |
| `qc_*` | 质控指标与统计 | 麻醉独有 |
| `ai_*` | AI 知识库 / 规则 / 调用日志 | 麻醉独有 |

## 与护理系统的共享边界

> 护理系统数据库权威字典：[`./护理系统新版本数据库文档.md`](./护理系统新版本数据库文档.md)
> 合并方案：[`../01_plan/merge-with-surgery-nursing.md`](../01_plan/merge-with-surgery-nursing.md)
> 集成迭代计划：[`../01_plan/phase-03-nursing-integration.md`](../01_plan/phase-03-nursing-integration.md)

### 数据归属原则

1. 共享主数据 / 共享流程的"主表"以护理系统为唯一权威源（SoT）。本仓库不再在 `sys_*` / `surg_*` 中复刻同名主表，而是建立映射 / 视图 / 适配。
2. 护理主导的共享业务模块（**患者交接、安全核查、器械清点、风险评估**）由护理域开发与维护；麻醉端仅"读取 / 抽取 / 简单补录共享字段"，不重建主表，不另起一份业务流程。
3. 麻醉特有业务（麻醉术前/后访视、麻醉用药、麻醉记录、PACU、麻醉小结、质控）由 `anes_*` 维护，与护理同名概念**字段独立**。
4. 共享表如缺字段，**优先在护理系统主表内补字段**；如属麻醉特有视角，再在本仓库以 `anes_*_ext`（外键指向护理主键）扩展。

### 表前缀的边界含义（更新）

- `sys_*` / `surg_*`：与护理共用，**护理为 SoT**；本仓库新增字段需先评估在护理主表补充。
- `anes_*`：麻醉独有；同时是承接 `*_ext` 扩展（外键指向护理主键）的位置。
- `anes_safety_check*`、`anes_safety_check_sign`：保留表名，但在合并后**仅作为对护理 `security`/`security_vice` 的适配层**，不再承载主数据。
- `mr_*`：文书中心共用，最终回写到护理 `emr`。
- `bill_*` / `qc_*` / `ai_*`：麻醉独有，不与护理重叠。

### 麻醉特有字段丰富（Phase 3-C 落地）

落地 migration：[`../../backend-tp6/database/migrations/2026_05_07_000001_phase3c_anes_field_enrichment.sql`](../../backend-tp6/database/migrations/2026_05_07_000001_phase3c_anes_field_enrichment.sql)

#### 命名约定

- 凡引用护理 SoT 主键的字段一律以 `nursing_` 前缀命名，且不建立物理外键（跨库 / 异库部署友好）。
- 麻醉端新建表统一用 `anes_*` 前缀；与护理同名概念字段独立，不共表。
- `anes_drug` 是规划文档里的别名；**实际表名为 `anes_medication`**，所有代码、API、迁移均以 `anes_medication` 为准。如需更细的"撤销/审核单"明细可后续追加 `anes_drug_record`，但本期不引入新表。
