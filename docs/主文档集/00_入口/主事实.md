生成时间：2026-06-02
用途：作为后续 AI 协作与开发人员的“事实唯一参照”。任何回答冲突时，先以本文件为准。

## 1. 当前系统定位

- 系统名：**手术麻醉管理系统（SAMIS）**，术护品牌下与“手术护理系统”并行的麻醉业务域。
- 形态：手术护理系统的麻醉应用扩展，**不是独立系统，不是另一套 HIS**。
- 目标范围：术前 / 术中 / 术后 / PACU / 麻醉质控的全周期闭环。
- 二期方向：设备采集、26 项国家质控指标、动态大屏、AI 辅助决策。

## 3. 手术麻醉系统与手术护理系统的关系

- 共底座：用户 / 角色 / 权限 / 菜单 / 字典 / 日志 / 人员 / 患者 / 手术通知 / 手术间 / 排班 / 输血 / 交接 / 安全核查 / 评估 / 器械清点等基础与共用业务，由护理 SoT 维护。
- 分业务：麻醉端建设 anes_* / qc_* / bill_* / mr_* 麻醉特有业务表。
- 单向依赖：samis（麻醉）可调用 app（护理），app 禁止反向引用 samis。
- 数据：同服务器双库，麻醉端通过 `/api/nursing/<domain>/list|detail` 与 4 个集成网关访问护理数据。
- 前端：麻醉前端调用护理登录（`POST /api/pc/v1/admin/login`），麻醉业务接口前缀 `/api/samis/pc/v1/`。

## 4. 麻醉记录单是核心专项，不要轻易改动已完成结构

- 位置：e:/code/麻醉记录单（独立 Vue3 + Vite 项目，无 UI 库依赖）
- 状态：FEATURES 列出的能力全部已实现（含 CHANGES 引入的异常预警与修改留痕）
- 测试：node:test 测试通过；打印样式有专项测试
- 红线：
  - 不要为了风格统一删改已经稳定的 9 Tab 主结构
  - 不要破坏现有数据模型（record / vitals / medications / infusions / transfusions / outputs / events / recovery / signatures / rescue / device）
  - 不要重写打印样式，已有打印修复结果（顶/底边线、时间刻度遮挡修复）必须保留
  - 后端接入时，按现有字段结构对接后端 API，而不是改记录单的内部数据模型

## 8. 已确认的关键事实清单（防止反复确认）

1. 麻醉后端最终落地形态：huliTp6System-master/samis/，API 前缀 `/api/samis/pc/v1/`。
2. 麻醉前端风格：参考护理前端，统一壳；记录单专项以业务子模块形式嵌入。
3. 数据库：同服务器双库（护理库 + samis 库），通过引用键关联。
4. 患者主数据：以护理 operatenotice 为 SoT，麻醉端不另起患者主表。
5. 安全核查、交接、器械清点、输血、风险评估：护理 SoT，麻醉端仅读 + 必要补签。
6. 麻醉记录、麻醉访视、麻醉用药、PACU、麻醉小结、质控：麻醉独立，与护理同名概念字段独立。
7. 权限编码：模块:资源:动作（例 `anes:record:edit`），与 docs/03_dictionary/permission-codes.md 一致。
8. 状态机：手术 / 文书 / PACU / 计费 / 签名共 5 套，以 docs/03_dictionary/status-machine.md 为准。
9. 已实现 API 清单：以 docs/04_delivery/current-implemented-apis.md 为准。
10. 已实现表清单：以 docs/04_delivery/current-database-dictionary.md + Phase migration 文件为准。
