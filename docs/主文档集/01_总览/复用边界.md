# 与手术护理系统复用边界说明

生成时间：2026-06-02
依据：
- 主源：docs/03_dictionary/database-tables.md、docs/05_adr/0001-nursing-sot-integration.md、docs/04_delivery/current-implemented-apis.md
- 现状：huliTp6System-master 已有 49 张表 + 25 个 DDD 业务域；samis 应用骨架已建（含 4 个只读集成网关）
- 决策：mazui/项目说明.txt 明确“麻醉模块以应用形式并入护理后端 samis/，API 前缀 /api/samis/pc/v1/”

> 核心立场：
> 手术麻醉系统不是另起一套 HIS，也不是把手术护理系统重写一遍。
> 它是在已有手术护理系统基础上，扩展出麻醉业务闭环。
> 共用基础平台，复用主数据，麻醉只新增麻醉特有业务。

---

## A. 可直接复用手术护理系统的能力（不重建）

| 能力 | 护理已实现位置 | 麻醉端使用方式 |
|---|---|---|
| 用户登录 | huliTp6 `power` domain / 护理 `POST /api/pc/v1/admin/login` | 麻醉前端直接调用护理登录接口 |
| 人员管理（医 / 麻 / 护） | `user`、`adminuser` domain + `user_information` 表 | 仅读，麻醉端通过 sys_user 适配层映射 |
| 科室管理 | `department` domain | 仅读 |
| 角色权限 | `power` domain + `powerlist`、`admin_user_group`、`admin_category` | 与 sys_role / sys_permission 对齐，不重做权限模型 |
| 菜单权限 | `power` domain | 麻醉菜单挂在统一菜单树下，由护理后端下发 |
| 字典管理 | `dict`、`options` domain | 麻醉端通过 sys_dict_* 适配，复用基础字典 |
| 操作日志 | 中间件 + `legacy` | 沿用同一审计中间件 |
| 登录日志 | 护理已实现 | 不另起 |
| 患者基础信息 | `operationinfo` domain（含住院号、姓名、性别等） | 仅读，必要时建患者快照 |
| 手术通知 / 排班基础数据 | `operationinfo`、`schedule` domain（`operatenotice`、`operatenotice_ext`） | 仅读 + 引用 OPERATIONID |
| 手术间信息 | `room` domain（`operation_room`、`operation_room_type`） | 仅读 |
| 医护人员信息 | `user_information` | 仅读 |
| 基础审计能力 | 已有中间件 | 直接复用 |
| 文件 / 打印基础能力 | 护理已有 `signature` 等 | 复用 |
| 系统配置 | `options` domain | 复用 |

要点：
- 上述能力**不在麻醉数据库内再建一份主表**。
- 麻醉端通过 `/api/nursing/<domain>/list|detail` 与 4 个集成网关（HuliBloodGateway / HuliHandoverGateway / HuliOperationGateway / HuliSecurityGateway）访问。
- 麻醉前端**禁止**直接对接护理业务表，只能走护理 SoT 聚合接口。

## B. 需要在护理系统基础上扩展的能力

这些能力护理已有“同名概念”，但麻醉视角与流程不同，需新增麻醉专用字段或表，并保持与护理 SoT 关联。

| 能力 | 与护理的关系 | 麻醉端处理 |
|---|---|---|
| 麻醉专用角色与权限 | 共用权限模型 | 在 sys_role / sys_permission 中新增 `anes:*` 权限点（详见 permission-codes.md） |
| 麻醉医生工作台 | 与护理工作台并行 | 在统一菜单下新增麻醉工作台分组 |
| 麻醉记录单 | 与护理交接 / 核查不同 | 全新数据结构（anes_record + 子表） |
| 麻醉事件字典 | 通用字典基础上扩展 | 在 sys_dict 中新增 `anes_event_type` 分类 |
| 麻醉用药字典 | 通用药品字典基础上扩展 | 新增 `anes_drug` 分类（含高警示标记） |
| 麻醉质控指标 | 护理无对应 | 新增 qc_* 表 |
| PACU 数据承接 | 复用患者 / 手术 / 人员 | 新增 anes_pacu_* 表，关联手术与患者 |
| 设备采集数据接入 | 护理无对应 | 新增 anes_device、原始消息表 |
| 麻醉小结与术后随访 | 与护理 after_operation_visit 字段独立 | 新增 anes_post_visit、anes_post_complication 等 |
| 麻醉术前访视 | 与护理 before_operation_visit 字段独立 | 新增 anes_pre_visit + anes_visit_item |
| 输血 / 交接 / 安全核查 | 护理是 SoT（唯一权威） | 麻醉只读 + 补签 + 字段抽取，不重建主表 |

## D. 表前缀复用约定（与 database-tables.md 一致）

| 前缀 | 含义 | 复用归属 |
|---|---|---|
| `sys_*` | 平台基础（用户/角色/权限/菜单/字典/审计/接口配置） | 共用 |
| `surg_*` | 手术主数据（手术 / 房间 / 排台 / 人员 / 时间 / 状态日志） | 共用 |
| `mr_*` | 电子病历 / 文书中心 | 共用 |
| `anes_*` | 麻醉业务（记录单 / 体征 / 用药 / 事件 / 抢救 / PACU / 访视 / 操作扩展） | 麻醉独有 |
| `bill_*` | 计费 | 麻醉独有 |
| `qc_*` | 质控指标与统计 | 麻醉独有 |
| `ai_*` | AI 知识库 / 规则 / 调用日志 | 麻醉独有（二期） |
| 护理原生表 | `operatenotice`、`security`、`apparatus*`、`handover_*`、`blood_*`、`evaluation_*` 等 | 护理 SoT，麻醉端只读 |

## E. 关键关联键

所有 `anes_*` 业务表必须保留如下外键（来自护理 SoT 或共用主数据）：

- `operation_id` ← 护理 `operatenotice.OPERATIONID`，或共用 `surg_operation.id`
- `patient_id` ← 共用 `surg_patient.id`（与护理 `operatenotice` 字段映射）
- `operation_room_id` ← 共用 `surg_room.id`（来自护理 `operation_room`）
- `staff_id` ← 共用 `sys_user.id`（与护理 `user_information` 映射）

新增表 `anes_nursing_shared_ref` 用于统一登记 reference / snapshot / sign-on / extract 的引用关系（Phase 3-C 已建）。

## F. 不要重复做的清单（红线）

> 这些事项一旦发现重复建设，必须停手退回护理 SoT。

1. 不要再建 `anes_patient` / `samis_patient` 这类患者主表。
2. 不要再建 `anes_security_check` 这类安全核查主表（之前规划过，已降级为映射）。
3. 不要再建 `anes_apparatus*` 器械清点表。
4. 不要再建 `anes_handover_*` 交接主表。
5. 不要再建 `anes_blood_*` 输血主表（仅可建 anes_record 内的输血明细，与护理输血主表通过引用关联）。
6. 不要在麻醉端复刻护理评估表（低体温 / 压力损伤 / 风险 / 血栓评分），只读引用即可。
7. 不要在麻醉端另起一套登录 / 权限 / 菜单 / 字典体系。
8. 不要为麻醉端做单独的患者管理界面，直接走护理已有页面或聚合查询。

## G. 结论

- 麻醉系统是**护理系统的麻醉应用扩展**，不是平行系统。
- 后端以护理 huliTp6System-master 为底座，麻醉应用挂在 `huliTp6System-master/samis/`，API 前缀 `/api/samis/pc/v1/`。
- 前端风格统一参考护理前端，记录单专项作为内嵌业务模块。
- 数据库可与护理同服务器双库，麻醉端只读引用护理主表，新增 `anes_*` / `qc_*` / `bill_*` 业务表。
- 任何“看起来护理已经做过”的能力，先查复用边界，再决定是否扩展或独立建设。
