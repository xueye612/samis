# 麻醉记录单后端接口要求

> **联调主文档（字段对照 / payload / 冲突契约）**：见 [麻醉记录单后端联调字段对照表.md](./麻醉记录单后端联调字段对照表.md)

## 风格约定

参考现有护理/samis 接口：

```text
GET /api-samis/pc/v1/operationInfo/getOperationInfo
```

麻醉模块统一前缀：

```text
/api-samis/pc/v1/anesthesiaRecord/*
/api-samis/pc/v1/anesthesiaSync/*
/api-samis/pc/v1/anesthesiaDevice/*
/api-samis/pc/v1/anesthesiaDict/*
```

**不再**把 `/api/anesthesia/records` 或 `/api/anes/*` 作为 samisweb 当前主接口。

## mock 与真实切换

| 项 | 说明 |
|---|---|
| 当前 | `backendFetch` mock（`src/services/mockApi.ts`） |
| 环境变量 | `VITE_ANESTHESIA_USE_MOCK`（默认 true） |
| 切换 | 设为 `false` 且后端可用时，同一 service 改走真实 HTTP，页面无感 |
| 前端禁止 | 直连 OceanBase；链路为 前端 → IndexedDB/队列 → API → DB |

## 初始化：手术通知单详情

```text
GET /api-samis/pc/v1/operationInfo/getOperationInfo?operationId=...
```

用途：

- 初始化患者信息、手术信息
- 写入 IndexedDB `snapshots`
- 获取 `operation_id` / `patient_id` / `room_id`
- 打印不依赖实时 operationInfo

#### 用药保存字段（saveMedication / batchSaveMedications）

```http
POST /api-samis/pc/v1/anesthesiaRecord/saveMedication
POST /api-samis/pc/v1/anesthesiaRecord/batchSaveMedications
```

请求体（camelCase 示例）须支持：

| 字段 | 说明 |
|---|---|
| `mode` | `single` / `intermittent` / `continuous` |
| `isSpecial` | **医生最终确认**；不得仅凭 drugName 强制覆盖 |
| `specialNo` | 特殊用药序号（可前端生成） |
| `specialCategory` | 特殊用药分类 |
| `specialReason` | 辅助及特殊用药区说明 |
| `eventTime` | 单次/间断事件时间 |
| `startTime` / `endTime` | 持续区间 |
| `dose` / `doseUnit` / `rate` / `rateUnit` / `route` | 剂量与途径 |

后端缺省规则：仅当请求**未传** `isSpecial` 时，可按药品字典 `defaultIsSpecial` 补齐；若前端显式传入 `isSpecial`，必须原样保存。

药品字典接口见 [麻醉药品字典接口要求.md](./麻醉药品字典接口要求.md)。显示规则见 [麻醉记录单用药显示规则.md](./麻醉记录单用药显示规则.md)。

前端「用药记录」弹窗仅调整录入 UI，保存字段与上述接口一致，不改变 `mode` / `isSpecial` 语义。

### 3. 同步

| 接口 | 用途 |
|---|---|
| pushBatch | 批量上传队列 |
| getSyncStatus | 同步状态 |
| resolveConflict | 冲突处理 |

队列字段（`sync_queue`）：

| 字段 | 说明 |
|---|---|
| local_id | 前端实体本地 ID |
| server_id | 后端返回后回填 |
| baseSyncVersion | 上传前实体版本 |
| syncVersion | 当前版本 |
| entityType | vital_sign / medication / monitor_raw 等 |
| operationType | create / update / delete / lock |
| payload | 实体 JSON |

pushBatch 返回（mock 已实现，真实需对齐）：

```json
{
  "success": [{ "localId": "...", "serverId": 1, "syncVersion": 2 }],
  "failed": [{ "localId": "...", "errorCode": "...", "message": "..." }],
  "conflicts": [{
    "localId": "...",
    "conflictType": "version_mismatch",
    "localPayload": {},
    "serverPayload": {},
    "localSyncVersion": 2,
    "serverSyncVersion": 3
  }]
}
```

### 3. 设备

| 接口 | 用途 |
|---|---|
| batchPushMonitorData | 监护仪 raw 批量 |
| batchPushVentilatorData | 呼吸机 raw 批量 |
| getLatestDeviceData | 拉取最新设备数据 |

**当前阶段结论（联调约定）**：

```text
前端上传 raw + vital_signs；
后端先原样保存；
后续真实设备接入后，再决定是否由后端统一抽样生成 vital_signs。
```

| 问题 | 当前结论 |
|---|---|
| 后端是否生成 vital_signs | 暂不强制，可先原样存前端 vital |
| 前端是否只上传 raw | 否，当前 raw + vital 均入队 |
| 幂等 | operation_id + entity_type + local_id |
| 重复上传 | 同 local_id 覆盖或返回已有 server_id |
