# 麻醉记录单本地持久化设计

## 为什么需要本地持久化

麻醉记录单是基于时间轴的连续记录，包含高频生命体征、用药线段、液体/输血、事件符号等。当前 samisweb 原型在刷新后会丢失运行态数据，无法满足术中连续记录与断网场景。

## 为什么使用 IndexedDB

- 结构化、异步、容量大，适合高频写入与批量查询
- 支持事务，适合「写记录 + 写队列 + 写审计」的原子组合
- localStorage 仅保存轻量偏好（当前病例、页码等），不存完整记录单

## IndexedDB 与 localStorage 分工

| 存储 | 内容 |
|---|---|
| IndexedDB `samis_anesthesia_local_v1` | 完整病例、子表、同步队列、设备 raw |
| localStorage `samis.anesthesiaRecord.localState.v1` | 字典覆盖、UI draft（原有逻辑保留） |
| IndexedDB `settings` | 当前页码、最近打开记录 |

## 本地表结构

数据库：`samis_anesthesia_local_v1`（Dexie，见 `src/services/anesthesia/localDb.ts`）

| 表 | 说明 |
|---|---|
| records | 主记录 + `case_payload` 全量 JSON |
| snapshots | 打印/锁定快照 |
| timeline_events | 事件 |
| medications / fluids / transfusions | 用药、液体、输血 |
| vital_signs | 趋势图/打印显示点 |
| monitor_raw / ventilator_raw | 设备原始数据 |
| io_records / lab_results | 出入量、血气 |
| audit_logs | 本地审计 |
| sync_queue | 待上传队列（仅前端） |
| settings | 页码等偏好 |

## local_id 与 server_id

- 前端生成 `local_id`（如 `case-001`、`vital-173...`）
- 后端返回 `server_id` 后写入对应行与 `records.server_id`
- 幂等键：`operation_id + entity_type + local_id`

## Hydration 状态

store 暴露三个标志（见 `src/stores/anesthesia.ts`）：

| 标志 | 含义 |
|---|---|
| `localPersistenceReady` | IndexedDB 初始化完成 |
| `isHydrating` | 正在从 IndexedDB 读取并合并数据 |
| `hasRestoredLocalData` | 至少合并过一条本地记录 |

流程：

```text
页面进入 / app 启动
  → isHydrating = true
  → 读取 records / settings / sync_queue / conflicts
  → 合并本地 case 覆盖 seed（仅本地有记录时）
  → restoreCasePageNo（页码在记录单渲染前完成）
  → isHydrating = false, localPersistenceReady = true
  → 渲染 LiveAnesthesiaSheet
```

`main.ts` 在 `app.mount()` 前 `await bootstrapAnesthesiaLocalPersistence()`，避免先渲染 seed 再闪变本地数据。

## 刷新恢复逻辑

1. 应用启动：`bootstrapAnesthesiaLocalPersistence()` 从 IndexedDB 合并；**无本地记录时才保留 seed mock**
2. hydration 完成前：`AnesthesiaRecord.vue` 显示 `--record-sheet-skeleton-height` 骨架，不渲染完整记录单主体
3. 打开记录单：`restoreCasePageNo(caseId)` 在 `sheetReady` 为 true 前完成
4. 优先使用本地 `case_payload` 覆盖 seed 中同 id 病例
5. 子表用于同步与设备追溯；UI 主渲染仍来自 `SurgeryCase`

## 页面刷新防抖动

- 不在 hydration 前渲染 `LiveAnesthesiaSheet` 与趋势图
- 顶部工具区固定约 68px 两行高度（临床行 + 辅助行），同步 chip 仅在异常态改变视觉权重，正常态不撑高布局
- 设备状态恢复不触发 `syncRecordDocument` 全量重绘
- 切换病例时短暂 `caseSheetReady = false`，避免旧数据闪屏
- 骨架高度使用 CSS 变量 `--record-sheet-skeleton-height`，随 `sheetZoom` 与多页数微调，减少 hydration 完成时高度跳变

## 顶部状态栏

- 组件：`RecordWorkstationTopbar.vue`（单行）+ `RecordStatusBar.vue`（inline 同步摘要）
- **单行布局**：左（返回、标题、患者、状态、冲突）| 中（缩放、适宽、病例切换、**日期时间**、**当前操作者**）| 右（同步摘要、保存/抢救/签名/打印/更多）
- 日期时间为本地时钟（每分钟刷新）；操作者与全局 `currentDoctorName` 一致
- **纸面快捷条** `RecordSheetQuickStrip.vue`：录入（给药/液体/输血/体征/出入量）+ 事件（低血压/升压药/手术开始/结束/插管/拔管/更多事件）
- 录入/事件按钮受 `recordActionRules` + `recordQuickActions` 状态规则控制
- 打印/打印预览：`@media print` + `.no-print` 隐藏全部顶部、侧栏与快捷条

## 操作按钮分层（2026-06-02）

| 层级 | 位置 | 内容 |
|---|---|---|
| 一级高频 | 顶部主操作 + 纸面快捷条 | 保存、抢救、签名、打印、给药、体征、液体、输血、出入量、关键事件 |
| 二级常用 | 右侧「快捷录入」+ 术中辅助 + 设备摘要 | 泵停、设备启停、同步设备、质控、阶段推荐事件、待补字段 |
| 低频 | 顶部「更多」+ 折叠面板 | 纸面显示设置、患者队列、质控侧栏、方案/麻醉方式、模拟设置 |

规则实现：`recordActionRules.ts`（顶部主按钮）、`recordQuickActions.ts`（纸面/侧栏录入与事件）

## 记录状态与按钮展示规则

规则实现：`src/services/anesthesia/recordActionRules.ts`

| 阶段 | 主操作 | 纸面录入/事件 | 顶部主按钮 |
|---|---|---|---|
| 未开始 | 启动记录 | 全部禁用 | 打印 |
| 采集中/记录中 | 状态 chip | 全部可用 | 保存、抢救、签名、打印 |
| 抢救中 | 抢救中 | 录入可用 | 保存、退出抢救、签名 |
| 已锁定 | 已锁定 | 全部禁用 | 申请修改、打印 |
| 已签名 | 已签名 | 全部禁用 | 申请修改、打印（无重复签名） |

已锁定/已签名状态下快捷录入与纸面编辑均禁用；设备控制仅在记录中/采集中/抢救中可用。

## 右侧设备与同步面板

- 组件：`RecordDeviceWorkbenchPanel.vue`（折叠项「设备与同步（详情）」，**默认折叠**）
- **仅设备运行摘要 + 详情入口**：监护仪/呼吸机运行状态（行内启动/停止）、最近采集、采样频率只读 chip
- 同步详情 / 冲突入口为文本按钮；失败时显示「同步失败，点击查看」
- 模拟模式、异常类型、可编辑采样间隔折叠在「模拟设置」`<details>` 内
- 核心区域高度约 160px 内，不使用 2×3 大卡片
- 不与顶部重复展示待上传/网络/采集状态/模拟模式摘要
- 打印/打印预览：随 `.record-toolbox` 隐藏

## 开发测试按钮

- 「注入测试冲突」仅在 `import.meta.env.DEV && VITE_ANESTHESIA_USE_MOCK` 时显示
- 生产/演示构建默认隐藏，不出现在临床操作界面

## 全局 bootstrap 门控

- `main.ts`：`app.mount()` 前 `await bootstrapAnesthesiaLocalPersistence()`
- `router.beforeEach`：导航前再次确保 bootstrap 完成（幂等）
- `AppLayout`：`localPersistenceReady && !isHydrating` 前显示轻量 loading，避免业务页误读 seed
- composable：`useAnesthesiaPersistenceGate()` / `ensureAnesthesiaPersistenceReady()`
- **原则**：IndexedDB 有数据 → 合并覆盖 seed；无本地数据 → seed 作为演示 fallback，不作为 hydration 前“真实数据”展示

## 自动保存

- 入口：`store.afterRecordMutation(caseId)` → `schedulePersistCase()` 防抖 1s
- 每次保存：写 records + 子表 + sync_queue（除非 `skipQueue`）
- 页码变更：`livePageNo` watch → `setRecordPageDraft` + 持久化

## 手动保存

「保存草稿」仍写 localStorage UI draft；临床数据通过 IndexedDB 自动保存。

## 锁定/打印

`printAndLockRecord` 触发 `operationType: lock` 入队，快照写入 `snapshots`。

## 清理策略

实现：`src/services/anesthesia/anesthesiaLocalCleanupService.ts`

| 函数 | 作用 |
|---|---|
| `cleanupSyncedDeviceRaw()` | 清理非当前记录、已同步、超过 7 天的 raw |
| `cleanupOldSuccessQueue()` | 清理超过 3 天的 success 队列 |
| `getLocalStorageStats()` | 统计 raw / queue 数量 |
| `runConservativeLocalCleanup()` | 手动触发保守清理 |
| `runStartupLocalCleanupIfDue()` | 启动后每 6 小时最多执行一次 |

保守规则：

- **不清理**：`sync_status !== success` 的 raw、`pending/failed/conflict/uploading` 队列、当前打开记录全部 raw
- **可清理**：非当前记录 + 已同步 + 超过 7 天的 `monitor_raw` / `ventilator_raw`；超过 3 天且不在当前记录最近 200 条内的 success 队列
- 清理结果写入 `audit_logs`（action=`local_cleanup`）

## 性能注意

- 防抖 1s，避免每次按键写库
- 设备 raw 与 vital_signs 分离；趋势图只读 display 点
- 批量上传设备数据，不每条立即请求后端

## 跨页 vital_signs 显示规则

实现：`recordPaginationEngine.isTimeOnPage` + `LiveAnesthesiaSheet.pageVitals` + `selectDisplayVitalsForBand`

| 规则 | 说明 |
|---|---|
| 页内归属 | 非末页：`pageStartTime <= measure_time < pageEndTime` |
| 末页 | `pageStartTime <= measure_time <= pageEndTime`（包含轴终点） |

# 麻醉记录单实时同步设计

## 与手术通知单 / 手术间接口的边界

- `operationInfo`、`room` 可先接真实接口（模块开关 `VITE_USE_REAL_OPERATION_INFO` / `VITE_USE_REAL_ROOM`）。
- 打开记录单：真实 `getOperationInfo` 仅合并**患者/手术/人员/手术间**等通知单字段，并写入 `snapshots`；**不**覆盖 vitals/medications/fluids 等 IndexedDB 临床数据。
- 已打印或已锁定：禁止远程 `operationInfo` 覆盖 `recordSnapshot`（`shouldSkipRemoteOperationRefresh`）。
- `anesthesiaSync`、`anesthesiaDevice` 在 Apifox 正式化之前保持 mock；临床保存仍走 IndexedDB + `sync_queue`。

## 同步队列

表：`sync_queue`（IndexedDB）

字段：`queue_id, batch_no, entity_type, operation_type, api_path, payload, status, retry_count, next_retry_at`

状态：

| status | 含义 |
|---|---|
| pending | 待上传 |
| uploading | 批次上传中 |
| success | 已成功 |
| failed | 失败，等待重试 |
| conflict | 服务端冲突（mock 暂未模拟） |

## local_id / server_id 映射

上传成功后 `anesthesiaSyncService` 将 `server_id` 写回对应实体表与 queue 行。

## batch_no 规则

```text
front_YYYYMMDD_HHmmss_###
monitor_YYYYMMDD_HHmmss_###
ventilator_YYYYMMDD_HHmmss_###
```

## 实时上传策略

| 数据类型 | 策略 |
|---|---|
| 手工事件/用药/液体 | 入队后 ~1.5s 批量上传 |
| vital_sign 显示点 | ~8s 批量 |
| monitor_raw / ventilator_raw | ~15s 批量 |
| 扫描兜底 | 每 5s 扫描 pending |

## 断网补传

- `navigator.onLine` 监听 offline/online
- 离线：继续本地保存 + 队列 pending
- 恢复在线：立即 `flushAnesthesiaSyncNow()`

## 失败重试

指数退避：`5 * 2^retry_count` 秒，上限 300s，写入 `next_retry_at`

## 高频设备节流

设备 mock 每 3~5s 写 raw，但上传合并为 15s 批次，避免打满网络。

## 冲突处理

版本冲突、锁定冲突、人工修正生命体征冲突等**不会自动覆盖**，详见本文「同步冲突处理」章节。

- 冲突写入 `sync_conflicts` 表
- 队列项 status=`conflict` 停止自动重试
- 页面顶部显示冲突数量，打开冲突面板处理

## UI 状态区收口（2026-06-02）

| 区域 | 职责 | 交互 |
|---|---|---|
| 顶部 `RecordWorkstationTopbar` 临床行 | 返回、患者/手术、状态 tag、按状态展示的操作按钮 | 主操作随记录阶段变化；已锁定不显示保存/抢救/签名 |
| 顶部辅助行（左） | 缩放、病例切换、快捷事件 | 低频视图切换收进「更多」 |
| 顶部辅助行（右）`RecordStatusBar` inline | 同步摘要 | 仅异常态突出；点击 chip → 同步详情/冲突面板 |
| 右侧「记录辅助」 | 待补字段、质控、关键时间 | 设备与同步默认折叠在「设备与同步（详情）」 |
| 质控侧栏 | 异常/完整度/缺陷 | 默认收起，按需展开 |

- **同步成功**：不弹窗；仅更新最近同步时间、顶部 chip、待上传数量
- **同步失败**：顶部红色失败 chip + 右侧弱提示链接
- **冲突**：顶部 + 右侧按钮入口；不自动弹大面板（签名/提交前校验除外）
- **打印/预览**：隐藏状态栏、设备面板、同步/冲突入口、开发测试按钮

## success 队列清理

实现：`cleanupOldSuccessQueue()`（`anesthesiaLocalCleanupService.ts`）

| 规则 | 说明 |
|---|---|
| 不清理 | `pending` / `uploading` / `failed` / `conflict` |
| 保留 | 当前打开记录最近 **200** 条 success |
| 可清理 | 其他 success 且 `updated_at` 超过 **3 天** |
| 映射 | `local_id` / `server_id` 保存在实体表，清理 queue 不影响映射 |

触发方式：启动后低频自动（6h）或 `runConservativeLocalCleanup()` 手动。

## 幂等要求

每条 payload 含 `localId, syncVersion, operationId, recordLocalId`

# 麻醉记录单同步冲突处理设计

## 1. 设计目标

本地 IndexedDB 与服务器 OceanBase 同步时，**不能**简单本地覆盖服务器，也**不能**简单服务器覆盖本地。发生冲突时必须显式记录、提示用户并支持人工决策。

## 2. 同步元数据要求

### 2.1 本地实体行（所有业务表）

每条本地记录必须包含：

| 字段 | 说明 |
|---|---|
| local_id | 前端生成 UUID |
| server_id | 后端返回，可为空 |
| sync_version | 单调递增版本 |
| sync_status | local_only / pending / success / failed / conflict |
| updated_at | 最后更新时间 |
| deleted_at | 软删除时间（void 时不物理删） |
| void_reason | 作废原因 |

### 2.2 上传队列 sync_queue

| 字段 | 说明 |
|---|---|
| operation_type | create / update / delete / void / lock / print |
| entity_type | record / vital_sign / medication / ... |
| local_id | 实体 local_id |
| server_id | 已知 server_id |
| base_sync_version | 上传时所依据的本地版本 |
| payload | 完整上传体 |

## 3. 冲突表 sync_conflicts

| 字段 | 说明 |
|---|---|
| conflict_id | 主键 |
| local_payload | 本地快照 JSON |
| server_payload | 服务器快照 JSON |
| conflict_type | version_mismatch / record_locked / vital_corrected / ... |
| resolve_status | pending / resolved / ignored |
| resolve_action | use_server / keep_local_correction / manual_merge / ignore_local / retry_sync |

## 4. 同步流程

```text
本地变更 → 写 IndexedDB + 入队(baseSyncVersion)
  ↓
pushBatch 逐条校验
  ↓
success → 写回 server_id + sync_version
failed  → 指数退避重试
conflict → 写 sync_conflicts + 队列 status=conflict
  ↓
用户打开冲突面板 → 选择处理策略 → 重新保存/重试
```

## 5. 冲突判断规则

| 场景 | 策略 |
|---|---|
| baseSyncVersion < serverSyncVersion（用药/液体/输血/事件） | **conflict**，不自动覆盖 |
| 服务器 record locked/printed + 本地 update/void | **conflict** |
| vital is_corrected=true + 设备数据写入 | **conflict** / 本地拒绝覆盖 |
| monitor_raw / ventilator_raw 重复 local_id 或 collect_time | **dedup success**，追加为主 |
| delete 操作 | 传 void + deleted_at + void_reason，服务器软作废 |

## 6. 幂等要求（后端）

同一 `record_id + entity_type + local_id` 重复 pushBatch：

- 已存在 → 返回 success + 原 server_id（不重复插入）
- 版本冲突 → 返回 conflict + serverPayload + serverSyncVersion

## 7. pushBatch 返回要求

每条 item 必须返回：

```json
{
  "entityType": "medication",
  "localId": "med-001",
  "serverId": 10002,
  "status": "success|failed|conflict",
  "serverSyncVersion": 3,
  "conflictType": "version_mismatch",
  "serverPayload": {},
  "message": ""
}
```

## 8. 冲突处理动作

| 动作 | 行为 |
|---|---|
| 使用服务器版本 | 本地 case 应用 server_payload |
| 保留本地修正 | 以手工修正重新入队，sync_version+1 |
| 人工合并 | 合并 JSON 后保存并重试 |
| 忽略本地 | 丢弃本地 pending，采用服务器 |
| 重试同步 | 队列重新 pending，携带最新 baseSyncVersion |

## 9. 页面提示

记录单顶部显示「冲突 N」，点击进入 `SyncConflictPanel` 查看 local/server 对比并处理。

## 10. 验收方式

1. mock 服务器先写入 medication syncVersion=2
2. 本地 baseSyncVersion=1 上传 → 收到 conflict
3. sync_conflicts 有记录，顶部冲突数 +1
4. 选择「使用服务器版本」→ 冲突清零，本地数据与 server_payload 一致
5. 锁定记录后本地修改 → conflict_type=record_locked
6. 人工修正 vital 后启动监护仪模拟 → 不产生覆盖（conflict 或跳过写入）

## 11. 代码入口

- `src/services/anesthesia/anesthesiaSyncConflict.ts`
- `src/services/anesthesia/anesthesiaSyncService.ts`
- `src/components/anesthesia/record/SyncConflictPanel.vue`
- mock：`src/services/mockApi.ts` → `handleSamisSyncPushBatch`

监护会话绑定 **患者 + 手术 + 麻醉记录单 + 手术间 + 设备**，不绑定浏览器页面。设备数据始终写入启动监护时的 `record_local_id`（`monitoringSessionService.ts` + IndexedDB `settings.monitoring_session_v1`）。

| 场景 | 规则 |
|---|---|
| 页面刷新 | 不解除监护会话；已采集数据保留；mock **不自动**继续采集，提示是否恢复 |
| 切换患者 | 提示「数据仍归属原患者」；暂停 mock tick；新患者须单独启动监护 |
| 切回原患者 | 从本地库合并体征；恢复会话状态；可恢复 mock |
| 离开未切回 | 真实设备：数据继续归属原记录单；mock：暂停/超时停止 tick，数据不删 |
| 停止监护 | 正常结束采集，保留数据 |
| 撤销监护 | 须填原因，作废本次会话内设备采集体征（`status=voided`） |

- 恢复设备元数据时**不修改**患者信息

## 时间轴起点（显示 / 分页 / 定位）

实现：`recordPaginationEngine.resolveRecordAxisStart`、`anesthesiaRecordEngine.resolveRecordAxisReferenceIso`、`roundAxisStartTime`。

| 阶段 | 参考时刻 | 显示起点 |
|---|---|---|
| 未入室 | `scheduledStart ?? plannedStart`（手术通知单预计开台） | 向前取整到整点或半点（默认 30 分钟步长） |
| 已入室 | `roomInTime`（真实入室时间） | 同上取整 |

约束：

- **不**使用 `anesthesiaStart` / `actualStart` 替代上述参考（避免轴起点随麻醉开始漂移）
- 起点变化**只**更新 `recordDocument.timeAxisPages` 与页码定位；用药/液体/体征/事件 ISO 时间**不重写**
- 录入「入室」或关键时间「入室」节点后：`syncRecordDocument` → `focusRecordPageByTime(roomInTime)`，纸面定位到入室所在页
- 打印预览与页面共用同一套 `buildRecordPagination` 结果
- **抢救模式**：仅 `minorInterval` 1 分钟与采集频率变化；起点规则不变；退出抢救恢复 5 分钟刻度并定位到当前临床时间附近

录入默认时刻（`resolveRecordSheetNowIso`）：已入室后不低于取整后的轴起点；未入室时不 clamp 到未来排班时刻。

## 状态提示位置

- **顶部临床行**：记录状态 tag；设备采集中时弱提示「设备采集中」
- **顶部辅助行右侧** `RecordStatusBar` inline：同步摘要（待传、冲突、失败、离线、最近同步），**不**展示设备运行详情
- **右侧**「设备与同步（详情）」：监护仪/呼吸机运行状态、最近采集、采样频率 chip；同步/冲突为详情入口；**默认折叠**

- 设备启动/停止/采集**不弹大提示**；同步成功仅更新时间 chip
- 打印模式下隐藏状态栏、设备面板、开发测试按钮及所有运行态 UI

| `isSpecial` | **医生最终确认**；不得仅凭 drugName 强制覆盖 |
| `specialNo` | 特殊用药序号（可前端生成） |
| `specialCategory` | 特殊用药分类 |
| `specialReason` | 辅助及特殊用药区说明 |
| `eventTime` | 单次/间断事件时间 |
| `startTime` / `endTime` | 持续区间 |
| `dose` / `doseUnit` / `rate` / `rateUnit` / `route` | 剂量与途径 |

后端缺省规则：仅当请求**未传** `isSpecial` 时，可按药品字典 `defaultIsSpecial` 补齐；若前端显式传入 `isSpecial`，必须原样保存。

药品字典接口见 [麻醉药品字典接口要求.md](./麻醉药品字典接口要求.md)。显示规则见 [麻醉记录单用药显示规则.md](./麻醉记录单用药显示规则.md)。

前端「用药记录」弹窗仅调整录入 UI，保存字段与上述接口一致，不改变 `mode` / `isSpecial` 语义。

### 3. 同步

| 接口 | 用途 |
|---|---|
| pushBatch | 批量上传队列 |
| getSyncStatus | 同步状态 |
| resolveConflict | 冲突处理 |

队列字段（`sync_queue`）：

| 字段 | 说明 |
|---|---|
| local_id | 前端实体本地 ID |
| server_id | 后端返回后回填 |
| baseSyncVersion | 上传前实体版本 |
| syncVersion | 当前版本 |
| entityType | vital_sign / medication / monitor_raw 等 |
| operationType | create / update / delete / lock |
| payload | 实体 JSON |

pushBatch 返回（mock 已实现，真实需对齐）：

```json
{
  "success": [{ "localId": "...", "serverId": 1, "syncVersion": 2 }],
  "failed": [{ "localId": "...", "errorCode": "...", "message": "..." }],
  "conflicts": [{
    "localId": "...",
    "conflictType": "version_mismatch",
    "localPayload": {},
    "serverPayload": {},
    "localSyncVersion": 2,
    "serverSyncVersion": 3
  }]
}
```

### 3. 设备

| 接口 | 用途 |
|---|---|
| batchPushMonitorData | 监护仪 raw 批量 |
| batchPushVentilatorData | 呼吸机 raw 批量 |
| getLatestDeviceData | 拉取最新设备数据 |

**当前阶段结论（联调约定）**：

```text
前端上传 raw + vital_signs；
后端先原样保存；
后续真实设备接入后，再决定是否由后端统一抽样生成 vital_signs。
```

| 问题 | 当前结论 |
|---|---|
| 后端是否生成 vital_signs | 暂不强制，可先原样存前端 vital |
| 前端是否只上传 raw | 否，当前 raw + vital 均入队 |
| 幂等 | operation_id + entity_type + local_id |
| 重复上传 | 同 local_id 覆盖或返回已有 server_id |
