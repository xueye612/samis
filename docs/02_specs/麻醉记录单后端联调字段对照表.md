# 麻醉记录单后端联调字段对照表

更新时间：2026-06-02  
适用范围：samisweb 麻醉记录单前端 ↔ TP6 后端 ↔ OceanBase  
代码主源：`src/types/anesthesiaLocalDb.ts`、`src/services/anesthesia/anesthesiaSyncQueue.ts`、`src/api/anesthesiaSync.ts`

> 本文档只做**接口契约与字段对照**，不涉及 UI 或记录单主体改造。

---

## 1. 文档目的

联调前需明确：

1. 前端 `entityType` 全集；
2. 本地 IndexedDB 表与 OceanBase 表映射；
3. 上传 API 路径与批量策略；
4. `pushBatch` 请求/返回/冲突结构；
5. 各实体 payload 示例与必填字段；
6. 幂等键、冲突判断、打印/设备追溯边界；
7. mock 与真实后端切换方式。

## 2. 接口风格

统一前缀（与护理系统一致）：

```text
/api-samis/pc/v1/operationInfo/getOperationInfo
/api-samis/pc/v1/anesthesiaRecord/*
/api-samis/pc/v1/anesthesiaSync/*
/api-samis/pc/v1/anesthesiaDevice/*
/api-samis/pc/v1/anesthesiaDict/*
```

**禁止**改用 `/api/anesthesia/...` 或 `/api/anes/...` 作为 samisweb 主链路。

统一响应封装：

```json
{
  "code": 0,
  "msg": "success",
  "data": {},
  "request_id": "mock-173..."
}
```

`code !== 0` 时前端 `unwrapSamisResponse` 抛错，队列项进入 failed 重试。

| `transfusion` | `transfusions` | `anes_transfusion` | `/anesthesiaRecord/batchSaveTransfusions` | 是 | 是 | **是** | 是 | 否 | 是 | 否 |

| `ventilator_raw` | `ventilator_raw` | `anes_respiratory` 或 `anes_device_raw_message` | `/anesthesiaDevice/batchPushVentilatorData` | 是 | 是 | 否 | 否 | **是** | 否 | 否 |
| `io_record` | `io_records` | `anes_intake_output` / `anes_io_record` | `/anesthesiaRecord/saveIoRecord` | 可批量 | 是 | 部分 | 是 | 否 | 是 | 否 |
| `lab_result` | `lab_results` | `anes_lab_result` | `/anesthesiaRecord/saveLabResult` | 可批量 | 是 | 部分 | 是 | 否 | 是 | 否 |
| `audit_log` | `audit_logs` | `anes_audit_log` | 建议 `/anesthesiaRecord/saveAuditLog`（待后端） | 是 | 是 | 否 | 否 | 否 | 否 | 否 |

- **冲突**：`medication` / `fluid` / `transfusion` / `timeline_event` 为前端 `CONFLICT_ENTITY_TYPES`，必须人工确认；`record` / `vital_sign` 在锁定、版本、人工修正场景也会冲突。
- **物理删除**：全实体统一 **void + `deleted_at`**，不做 DELETE。
- **当前前端实际上传**：已独立入队的有 `record`、`timeline_event`、`medication`、`vital_sign`、`monitor_raw`、`ventilator_raw`；`fluid` / `transfusion` / `io_record` / `lab_result` 已写入 IndexedDB 子表，多数仍随 `saveRecord` 全量更新入队，**联调时需后端同时支持 pushBatch 细粒度与 saveRecord 粗粒度**（见 §13）。

## 4. 统一同步字段（pushBatch item）

每个上传 item 必须包含：

```json
{
  "entityType": "medication",
  "operationType": "create",
  "localId": "med-1730000001",
  "serverId": null,
  "recordLocalId": "case-or04",
  "recordServerId": 10001,
  "operationId": "case-or04",
  "baseSyncVersion": 0,
  "localSyncVersion": 1,
  "apiPath": "/api-samis/pc/v1/anesthesiaRecord/saveMedication",
  "payload": {}
}
```

### 4.1 字段说明

| 字段 | 生成方 | 必填 | 用途 |
|---|---|:---:|:---|
| `entityType` | 前端 | 是 | 路由到 OceanBase 表与处理器 |
| `operationType` | 前端 | 是 | `create` / `update` / `delete` / `void` / `lock` / `print` |
| `localId` | **前端** | 是 | 本地 UUID；**幂等主键之一** |
| `serverId` | 后端 | 新增时为空 | 首次成功后回填本地 |
| `recordLocalId` | 前端 | 是 | IndexedDB `records.local_id` |
| `recordServerId` | 后端 | 否 | 首次 createRecord 后回填 |
| `operationId` | 前端/手术通知 | 是 | 手术排班 ID；**幂等主键之一** |
| `baseSyncVersion` | 前端 | 是 | 本次修改基于的服务端版本 |
| `localSyncVersion` | 前端 | 是 | 本地实体 `sync_version` |
| `apiPath` | 前端 | 是 | pushBatch 内路由到具体处理器 |
| `payload` | 前端 | 是 | 业务 JSON |

### 4.2 幂等键（后端必须实现）

```text
UNIQUE (operation_id, entity_type, local_id)
```

或等价：`operationId + entityType + localId`。

重复提交同一 localId：

- `create` 且已存在 → 返回原 `serverId` + 当前 `serverSyncVersion`（success，不重复插入）；
- `update` 且 `baseSyncVersion` 过期 → 返回 conflict。

### 4.3 冲突判断字段

| 场景 | 判断依据 |
|---|---|
| 版本冲突 | `baseSyncVersion < server.sync_version` |
| 记录已锁定 | `anes_record.status = locked` 且 operationType 为 update/void/delete |
| 记录已打印 | `anes_record.printed_at IS NOT NULL` 且本地尝试 update |
| 生命体征人工修正 | 服务端 `is_corrected=1` 且设备覆盖 |
| 同时间点重复 | `measure_time` + `record_id` + 指标桶冲突 → `duplicate_time_point` |
| 字段级冲突 | 关键字段 server/local 不一致 → `field_conflict` |

## 5. pushBatch 主同步接口

```text
POST /api-samis/pc/v1/anesthesiaSync/pushBatch
```

### 5.1 请求示例

```json
{
  "batchNo": "front_20260602_130000_001",
  "operationId": "case-or04",
  "recordLocalId": "case-or04",
  "recordServerId": 10001,
  "clientTime": "2026-06-02 13:00:00.000",
  "items": [
    {
      "entityType": "medication",
      "operationType": "create",
      "localId": "med-1730000001",
      "serverId": null,
      "baseSyncVersion": 0,
      "apiPath": "/api-samis/pc/v1/anesthesiaRecord/saveMedication",
      "payload": {
        "drugName": "丙泊酚",
        "dose": 120,
        "doseUnit": "mg",
        "route": "静脉",
        "mode": "单次",
        "startTime": "2026-06-02 10:10:00.000"
      }
    }
  ]
}
```

### 5.2 批次处理原则

1. **允许部分 success / failed / conflict**，禁止整批回滚；
2. 每条 item **必须**返回 `localId`；
3. 新增 success **必须**返回 `serverId`；
4. 更新 success **必须**返回 `serverSyncVersion`；
5. conflict **必须**返回 `conflictType` + `serverPayload`；
6. failed **必须**返回 `message`。

### 5.3 返回示例

```json
{
  "code": 0,
  "msg": "success",
  "data": {
    "batchNo": "front_20260602_130000_001",
    "results": [
      {
        "entityType": "medication",
        "localId": "med-1730000001",
        "serverId": 30001,
        "status": "success",
        "serverSyncVersion": 2,
        "message": ""
      },
      {
        "entityType": "vital_sign",
        "localId": "vital-1730000002",
        "serverId": 40001,
        "status": "conflict",
        "conflictType": "version_conflict",
        "serverSyncVersion": 4,
        "message": "服务器记录已被其他终端修改",
        "serverPayload": {
          "HR": 92,
          "SBP": 128,
          "source": "设备采集"
        }
      },
      {
        "entityType": "monitor_raw",
        "localId": "monitor-1730000003",
        "status": "failed",
        "message": "device channel unavailable"
      }
    ]
  },
  "request_id": "req-20260602-130001"
}
```

### 5.4 前端当前 mock 与真实差异

| 项 | mock（`mockApi.ts`） | 真实后端目标 |
|---|---|---|
| conflictType | `record_locked` / `record_printed` / `version_mismatch` 等 | 对齐 §6 统一枚举 + 映射 |
| 部分失败 | 已模拟 | 必须支持 |
| serverSyncVersion | mock 自增 | 来自 OceanBase 行版本 |

前端映射函数：`parseConflictTypeFromResult()`（`anesthesiaSyncConflict.ts`）需在联调时扩展枚举映射表。

## 6. 冲突返回结构

### 6.1 标准 conflict item

```json
{
  "entityType": "medication",
  "localId": "med-1730000001",
  "serverId": 30001,
  "status": "conflict",
  "conflictType": "version_conflict",
  "serverSyncVersion": 4,
  "message": "服务器记录已被其他终端修改",
  "serverPayload": {
    "drugName": "丙泊酚",
    "dose": 100,
    "doseUnit": "mg",
    "startTime": "2026-06-02 10:10:00.000"
  }
}
```

### 6.2 conflictType 枚举

| 后端标准值 | 前端当前值 | 含义 |
|---|---|---|
| `version_conflict` | `version_mismatch` / `server_newer` | 版本过期 |
| `server_locked` | `record_locked` | 服务端已锁定 |
| `deleted_remote` | （待扩展） | 服务端已 void |
| `duplicate_time_point` | （待扩展） | 同时间点重复 vital/event |
| `field_conflict` | `entity_conflict` | 字段级不一致 |
| — | `record_printed` | 已打印不可改 |
| — | `vital_corrected` | 人工修正不可被设备覆盖 |

### 6.3 冲突处理原则

1. **设备 raw**：以追加为主，一般不冲突；重复 localId 幂等返回 success；
2. **vital_sign**：人工修正 / `is_corrected` 优先于设备覆盖；
3. **medication / fluid / transfusion / timeline_event**：必须人工确认（前端 `SyncConflictPanel`）；
4. **已锁定记录**：本地 pending 不得直接覆盖服务端；
5. **删除**：统一 `operationType: void`，写 `deleted_at` / `void_reason`，禁止物理 DELETE。

冲突解决接口：

```text
POST /api-samis/pc/v1/anesthesiaSync/resolveConflict
```

## 8. 前端当前上传策略（联调约定）

1. **同时上传 raw + vital_signs**  
   - 监护仪/呼吸机 mock 每 4s（抢救 2s）写 raw；按 5min/1min 桶写 vital_signs。  
2. **raw** → 设备追溯、质控、后续算法；**不直接绘制趋势图**。  
3. **vital_signs** → 趋势图、监测带、**打印唯一体征来源**。  
4. **后端当前阶段**：raw 与 vital 均原样落库，不要求后端即时抽样。  
5. **后续真实设备接入**：可再评估是否改为「仅上传 raw、后端统一生成 vital_signs」。  
6. **local_id 幂等**：断网补传、重复 batch 不重复插入。  
7. **server_id 回填**：success 后写入 IndexedDB 对应行 + `sync_version` 更新。

设备上传频率（前端已实现）：

| 模式 | raw | vital 显示 |
|---|---|---|
| 正常模拟 | 4s | 5min |
| 异常模拟 | 4s | 5min（可控异常） |
| 抢救模式 | 2s | 1min |

## 12. 需后端确认的问题（TBD）

1. **operationId 格式**：当前 mock 使用 `case-or04` 同 `recordLocalId`；生产是否统一为 `OPERATIONID` 字符串？  
2. **recordStatus 枚举映射**：前端 `未开始/采集中/补记中/已完成/已锁定` ↔ 后端 `draft/recording/completed/locked` 对照表。  
3. **fluid/transfusion 入队策略**：何时从 `saveRecord` 粗粒度改为独立 `batchSaveFluids` / `batchSaveTransfusions` 入队。  
4. **raw 幂等键**：`localId` 唯一 vs `collectTime+deviceId` 去重，是否允许同一秒多条 raw。  
5. **ventilator_raw 落表**：独立 `anes_respiratory` 还是统一 `anes_device_raw_message`。  
6. **snapshot 版本**：每次 saveSnapshot 递增 version 还是覆盖；与 lock/print 关系。  
7. **audit_log 是否强制上传**：仅本地留存还是必须同步到 `anes_audit_log`。  
8. **conflictType 枚举最终版**：是否采用 §6.2 后端标准值，前端做映射层。  
9. **getRecordDetail 粒度**：一次返回全子表 vs 分页拉取 vitals/raw。  
10. **鉴权**：Token 与护理 PC 共用还是独立 scope；pushBatch 是否需要手术间权限校验。
