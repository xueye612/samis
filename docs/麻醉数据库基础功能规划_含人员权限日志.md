# 麻醉数据库基础功能规划（含人员、权限、登录日志、操作日志）

> 适用范围：手术麻醉管理系统 / 麻醉记录单 / PACU / 术前术后访视 / 麻醉质控 / 系统管理基础数据库规划。  
> 设计目标：在护理系统已有数据库能力之上，先完成麻醉专科必须新增的数据底座，避免重复建设手术通知单、人员、手术间、护理评估、交接、输血、电子病历附件等已存在能力；后续再逐步扩展设备采集、电子病历归档、质控指标、CA 签名、接口对接。

---

## 一、总体设计原则

麻醉数据库不能只围绕“麻醉记录单一张表”设计，而应围绕围术期麻醉业务全过程建立数据底座。

建议采用以下设计原则：

1. **一台手术一个主索引**：以手术 ID / OPERATIONID 贯穿术前、术中、PACU、术后、质控、文书归档。
2. **患者与手术信息保存快照**：打印、归档后的麻醉文书不能受 HIS 后续修改影响。
3. **麻醉记录单以时间轴为核心**：用药、液体、输血、事件、生命体征、出入量、血气都应绑定时间点或时间段。
4. **生命体征分层保存**：区分设备原始数据、记录单显示数据、人工修正数据。
5. **人员、权限、日志独立规划**：系统必须具备登录日志、操作日志、审计日志、打印日志、接口日志、异常日志。
6. **删除改为作废/撤销**：医疗文书和关键业务记录不建议物理删除。
7. **支持打印锁定和修改留痕**：打印后锁定，修改必须记录原因、修改人、修改前后内容。
8. **字典和模板可配置**：生命体征项目、事件符号、麻醉方式、药品、液体、评分、表单字段不应写死。
9. **兼容护理系统已有数据**：可关联护理系统的手术通知单、人员、手术间、安全核查、交接、输血、电子病历附件等基础能力。
10. **先核心闭环，后扩展**：第一版先建麻醉记录单核心表，再逐步扩展术前、PACU、术后、质控、设备采集。

## 二、数据库模块总览

建议麻醉数据库按以下 12 个业务域规划。

| 序号 | 业务域 | 主要内容 | 优先级 |
|---|---|---|---|
| 1 | 手术与患者主索引 | 手术索引、患者快照、手术快照 | P0 |
| 2 | 人员与组织权限 | 人员、角色、菜单、按钮、数据权限 | P0 |
| 3 | 登录与安全日志 | 登录日志、Token、会话、密码修改、账号锁定 | P0 |
| 4 | 麻醉记录单核心 | 记录主表、分页、事件、生命体征、用药、液体、输血、出入量 | P0 |
| 5 | 文书打印与审计 | 文书、打印、PDF、归档、签名、操作审计 | P0 |
| 6 | 术前麻醉管理 | 术前访视、麻醉评估、气道评估、麻醉计划、知情同意 | P1 |
| 7 | PACU 恢复室 | PACU 接收、床位、生命体征、评分、用药、转出 | P1 |
| 8 | 术后随访 | 疼痛、恶心呕吐、并发症、满意度、镇痛 | P1 |
| 9 | 质控管理 | 指标字典、质控结果、缺陷、整改、抽取日志 | P1 |
| 10 | 设备采集 | 设备字典、设备连接、原始消息、原始生命体征 | P2 |
| 11 | 配置字典 | 药品、液体、血制品、事件符号、监测项目、模板字段 | P0/P1 |
| 12 | 接口与归档 | HIS、LIS、输血、EMR、设备接口日志 | P2 |

### 4. 护理系统既有能力复用对照

护理系统已经覆盖了大量围手术期基础能力，麻醉库不建议全部重建。建议以 `OPERATIONID` 作为跨系统自然键，以 `anes_surgery_index.id` 作为麻醉库内部主键；对外同步时保留 `source_table`、`source_id`、`source_system`、`source_updated_at` 等字段，便于追溯来源。

| 护理系统已有表 | 已覆盖能力 | 麻醉库处理方式 | 关联到麻醉表 |
|---|---|---|---|
| `operatenotice` | 手术通知单、患者基本信息、手术信息、麻醉方式、手术间、人员、计划时间、入出室、PACU 出室、取消、锁定归档标识 | 作为手术主索引与快照来源，不直接替代麻醉记录单 | `anes_surgery_index`、`anes_patient_snapshot`、`anes_operation_snapshot`、`anes_record` |
| `operatenotice_ext` | 合并症、特殊感染、术前生命体征、特殊体位/设备、输血史、护士/麻醉备注 | 作为术前评估、风险评估和患者快照的补充来源 | `anes_patient_snapshot`、`anes_pre_visit`、`anes_risk_assessment` |
| `operation_room`、`operation_room_type` | 手术间、手术间类别、手术间分组 | 可同步为麻醉系统房间字典，避免单独维护两套手术间 | `sys_operation_room`、`anes_device` |
| `department_ward_info` | 科室、病区、楼层、护士站/医师办电话 | 可同步为科室组织基础数据 | `sys_dept`、`anes_patient_snapshot` |
| `user_information` | 姓名、工号、用户类型、护士类型、层级、科室、手机号、实习标识 | 可作为人员基础来源；登录密码需重构为不可逆哈希，角色权限独立建模 | `sys_user`、`sys_user_role`、`anes_signature` |
| `powerlist`、`admin_user_group` | 旧系统功能权限、用户组 | 可迁移菜单和角色初始数据；麻醉新增按钮/API/数据权限仍用新权限模型 | `sys_role`、`sys_menu`、`sys_permission`、`sys_data_scope` |
| `operation_pb_nurse` | 手术间排班、巡回/洗手/器械/麻醉人员安排 | 作为手术人员快照和麻醉排班来源 | `anes_operation_snapshot`、可扩展 `anes_case_staff` |
| `before_operation_visit`、`before_operation_visit_back` | 护理术前访视、取消访视原因、过敏史、手术史、皮肤、心理、感染、签名 | 不替代麻醉术前评估；可作为麻醉访视引用和前置信息 | `anes_pre_visit`、`anes_risk_assessment`、`anes_document` |
| `after_operation_visit` | 护理术后访视、皮肤、舒适度、签名、随访意见 | 可作为术后随访参考；麻醉术后随访需补充镇痛、神经、气道、并发症指标 | `anes_post_followup`、`anes_post_complication` |
| `handover_send`、`handover_receive_restore` | 手术交接、恢复室交接、带入管路、输液/血液、生命体征、皮肤、签名、PDF | PACU 接收/转出可复用交接信息；麻醉库补充 Aldrete/VAS/PONV/苏醒事件 | `anes_pacu_record`、`anes_pacu_receive`、`anes_pacu_transfer`、`anes_document` |
| `security`、`security_vice` | 手术安全核查、签名、文书上传 | 不重复建安全核查业务表；麻醉质控只引用核查完成状态 | `anes_quality_result`、`anes_document` |
| `apparatus` 系列表 | 器械清点、器械包、止血带、签名、PDF | 不属于麻醉核心库；仅在麻醉记录/质控里引用完成状态或止血带信息 | `anes_timeline_event`、`anes_procedure`、`anes_quality_result` |
| `medicant_enforcement`、`medicant_revoke` | 护理扫码/手工用药执行、撤销 | 可作为术前/术中护理执行用药来源；麻醉给药仍按剂量、泵速、途径结构化保存 | `anes_medication`、`anes_audit_log` |
| `blood_collection`、`blood_confirm` | 取血、输血记录、输血前/中/后生命体征、不良反应、核对签名、PDF | 可作为输血系统/护理输血来源；麻醉记录单保存展示所需输血时间轴和出入量 | `anes_transfusion`、`anes_intake_output`、`anes_vital_sign`、`anes_document` |
| `evaluation_hypothermy`、`evaluation_ressure_damage`、`evaluation_surgical_risk`、`evaluation_venous_thrombus` | 低体温、压力损伤、手术风险、VTE 评估及 PDF 上传 | 可作为风险/质控来源；麻醉库只补充麻醉相关风险和围术期事件 | `anes_risk_assessment`、`anes_quality_result`、`anes_document` |
| `emr` 以及各业务表的 `isupload`、`needupload`、`needcreat`、`pdfpath` | 电子病历附件、PDF/PNG 地址、平台抓取状态 | 麻醉文书归档可兼容此模式，但建议统一到版本化文书表 | `anes_document`、`anes_document_version`、`anes_emr_upload_log` |
| `option_cat`、`form_content_dictionary`、`form_content_dictionary_options` | 通用下拉、表单内容字典 | 可迁移为基础选项；麻醉专科模板、事件符号、生命体征图形仍需独立配置 | `anes_dict_category`、`anes_dict_item`、`anes_template_field` |

落地时建议把护理系统表分为三类：

1. **直接引用类**：手术通知单、手术间、科室、人员、护理安全核查、器械清点、护理交接、电子病历附件。
2. **同步快照类**：患者信息、手术信息、人员排班、术前基础生命体征、特殊感染、术后去向。
3. **麻醉专科新增类**：麻醉记录单时间轴、麻醉生命体征显示数据、设备原始数据、麻醉用药、麻醉机参数、麻醉小结、麻醉质控、医疗文书审计。

## 四、人员与组织信息

麻醉系统不能只保存“麻醉医生姓名”，应建立完整人员基础表，支持医生、麻醉医生、护士、管理员、实习/进修人员等角色。

### 1. 人员信息表：`sys_user`

作用：保存系统登录用户和临床人员基础信息。

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | bigint | 主键 |
| `user_code` | varchar(64) | 用户编码/院内人员编码 |
| `work_no` | varchar(64) | 工号 |
| `login_name` | varchar(64) | 登录名 |
| `password_hash` | varchar(255) | 密码哈希，不存明文 |
| `real_name` | varchar(64) | 姓名 |
| `pinyin_code` | varchar(64) | 拼音码 |
| `gender` | varchar(16) | 性别 |
| `mobile` | varchar(64) | 手机号 |
| `id_card` | varchar(64) | 身份证号，可脱敏 |
| `user_type` | varchar(32) | doctor/anesthetist/nurse/admin/technician |
| `professional_title` | varchar(64) | 职称 |
| `qualification_no` | varchar(128) | 执业资格编号 |
| `dept_code` | varchar(64) | 科室编码 |
| `dept_name` | varchar(128) | 科室名称 |
| `ward_code` | varchar(64) | 病区编码 |
| `ward_name` | varchar(128) | 病区名称 |
| `nurse_type` | varchar(32) | 手术室/本院/实习/进修等 |
| `nurse_level` | varchar(32) | 护士层级 |
| `anesthesia_group` | varchar(64) | 麻醉分组 |
| `is_internship` | tinyint | 是否实习/进修 |
| `status` | tinyint | 1启用 0停用 2锁定 |
| `last_login_time` | datetime | 最近登录时间 |
| `password_updated_at` | datetime | 密码更新时间 |
| `created_at` | datetime | 创建时间 |
| `updated_at` | datetime | 更新时间 |
| `is_deleted` | tinyint | 是否删除/作废 |

说明：

- 原护理系统中 `user_information` 已包含姓名、工号、类型、科室、手机号、是否实习等字段，麻醉系统可参考并规范化。
- 密码不要沿用明文或可逆字段，建议统一使用 `password_hash`。
- 临床签名可关联 `sys_user`，但签名记录应独立保存。
- `user_information.type` 可初步映射为 `doctor/anesthetist/nurse`，但麻醉护士、PACU 护士、质控员、接口账号等角色应通过 `sys_role` 分配，不建议只靠人员类型判断权限。
- 如第一版仍由护理系统统一登录，`sys_user` 可以先作为人员镜像表使用；真正启用麻醉独立登录后，再补齐 `password_hash`、`sys_session`、`sys_login_log`。

## 十九、后续设计注意事项

1. **用户表和人员签名不要混在一起**：用户基础信息在 `sys_user`，签名动作在 `anes_signature`。
2. **操作日志和审计日志要分开**：操作日志记录“做了什么”，审计日志记录“改了什么”。
3. **登录日志要记录失败原因**：便于排查账号锁定、密码错误、权限异常。
4. **所有医疗文书类表都应有 `is_deleted` 或 `status`**：不要物理删除。
5. **所有打印文书都应有 PDF 路径和版本号**。
6. **麻醉记录单核心数据必须结构化**：不能只存一大段 JSON 或图片。
7. **设备原始数据和记录单显示数据必须分开**。
8. **PACU 不要完全套用术中麻醉记录单**：PACU 重点是接收、观察、评分、转出。
9. **质控指标从结构化数据提取**：不要靠人工二次录入。
10. **字典和模板尽早设计**：避免后续不同医院、不同科室、不同麻醉方式无法扩展。

## 二十、最终结论

麻醉数据库基础规划应包含两条主线：

第一条是 **临床业务主线**：

```text
手术索引 → 术前评估 → 麻醉计划 → 麻醉记录单 → PACU → 术后随访 → 质控归档
```

第二条是 **系统治理主线**：

```text
人员 → 角色权限 → 登录日志 → 操作日志 → 审计日志 → 打印归档 → 接口日志
```

第一版最重要的是先把 `sys_user`、`sys_login_log`、`sys_operation_log`、`anes_record`、`anes_timeline_event`、`anes_vital_sign`、`anes_medication`、`anes_fluid`、`anes_transfusion`、`anes_intake_output`、`anes_document`、`anes_audit_log` 这些基础表建起来。

这样系统既能支撑 Vue 前端的麻醉记录单展示，也能支撑后续真正进入临床使用时需要的人员管理、权限控制、日志追溯、打印锁定、文书归档和质控提取。
